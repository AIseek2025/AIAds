## 1. 执行摘要

本报告按 gstack 所强调的 **结构化角色分工** 与 **端到端冲刺节奏**（Think → Plan → Build → Review → Test → Ship → Reflect），对 AIAds 下一阶段开发进行 **市场调研、产品规划、架构设计、开发拆解、测试与发布策略** 的系统化输出，并说明每一阶段如何 **借鉴 gstack 的“技能”逻辑**（不等同于在 Cursor 内真实执行 `/office-hours` 等 slash 命令，而是 **按其意图与产出物** 复刻流程）。

**核心结论**：

1. **产品再定义（gstack `/office-hours` 精神）**：需求文档描述的是「全链路 KOL 投放操作系统」；对当前阶段更稳妥的表述是 **「中小 KOL × 中国供应链广告主」的可信撮合与可审计结算**，先以 **CPM 透明化 + 订单/资金闭环 + 反作弊最小集** 为楔子，再扩展智能匹配、频次治理与全球化支付税务。
2. **工程现实（gstack `/plan-eng-review` 精神）**：AIAds 已具备 **Express + Prisma + React/Vite + 生产发布脚本** 的单体/模块化后端形态；文档中的「完整微服务 + Mongo + ES + Kafka」应作为 **演进目标**，而非下一迭代的强制交付，避免拖垮迭代速度。
3. **下一迭代建议（gstack `/ship` + `/qa` 精神）**：以 **可度量里程碑** 驱动：每迭代必须有 **可运行路径、可回滚发布、自动化健康检查、最小回归集**。

## 4. Phase A — 市场调研（gstack：`/office-hours`）

### 4.1 市场与需求现实

- **需求侧**：中国出海广告主需要 **可验证曝光** 与 **可控成本**；中小 KOL 需要 **确定性收入** 与 **低门槛接单**。
- **供给侧**：TikTok / YouTube / Instagram 等平台 API 能力、配额与合规差异大，**数据采集与归因**是长期成本中心。
- **竞争格局**：综合平台（大型 KOL 市场）、SaaS 工具、服务商代投并存；AIAds 差异化仍宜落在文档已写的 **「中小 KOL + 供应链 + 自动化」**，但需 **单点打透** 再扩展。

### 4.2 关键假设（可证伪）

1. **H1**：广告主愿为「可审计的 CPM + 透明结算」付平台佣金。  
2. **H2**：KOL 愿接受 **平台托管结算** + **KYC/税务** 渐进式引入。  
3. **H3**：早期客户更在意 **交付与资金安全**，而非「AI 匹配有多炫」。  

### 4.3 与需求文档的对齐

需求文档第 1–7 章（CPM、结算、检索、审核、频次、邀请、预算）覆盖 **完整操作系统**；按 gstack 的 **最窄楔子** 原则，建议先收敛为 **「可上线的信任闭环」**，再并行扩展 AI 与复杂治理。

## 5. Phase B — 产品规划（gstack：`/plan-ceo-review`）

### 5.1 北极星指标（建议）

- **业务**：单月 **有效计费曝光**（经反作弊最小集过滤）与 **成功结算笔数**。  
- **信任**：争议率、退款率、结算时效。  
- **效率**：从建单到 KOL 发布内容的中位时间。

### 5.2 范围决策（Hold Scope / Reduction）

- **本阶段做强**：订单与活动主流程、资金流水与对账、曝光与 CPM 的 **口径与展示**、基础反作弊规则、管理端审核最小闭环。  
- **本阶段刻意不做满**：全球税务全自动、全平台深度 API、完整 ES 搜索、全量 Kafka 事件流（与文档第八章「理想态」区分）。

### 5.3 MVP 切片（3 个可交付增量）

1. **MVP-1 透明**：广告主/KOL 侧能看到 **同一口径** 的曝光与应付/应收，并明确 **结算窗口**（文档第 1、2 章）。  
2. **MVP-2 可信**：**冻结预算 + 超额保护**（文档第 7 章）+ 基础风控（文档第 4、5 章的简化版）。  
3. **MVP-3 增长**：KOL 入驻与 **可解释匹配**（文档第 3 章的简化版：规则 + 特征，而非大模型一切）。

## 6. Phase C — 架构设计（gstack：`/plan-eng-review`）

### 6.1 当前基线（与代码仓库一致）

- **后端**：Node.js + Express + TypeScript + Prisma + PostgreSQL。  
- **前端**：React + Vite + MUI。  
- **部署**：Nginx 静态资源 + PM2 进程 + 发布脚本（见 `DEPLOY_PROD.md`）。

### 6.2 与需求文档第八章「微服务理想态」的映射

| 文档组件 | 近期落地（建议） | 演进说明 |
|----------|------------------|----------|
| API Gateway | Nginx 反向代理 + 鉴权中间件 | 后续可拆网关 |
| 匹配服务 | 模块化 `services` + 队列异步（若引入） | 先同进程，后拆 |
| 结算服务 | 账务流水 + 定时任务 + 幂等键 | 对齐 Stripe/PayPal 接入节奏 |
| 数据分析 | PostgreSQL 聚合 + 物化视图/批任务 | ES 后再上 |
| 消息队列 | Redis 或轻量队列优先 | Kafka 为规模化阶段 |

### 6.3 数据与事件（最小可行）

- **强一致**：订单、支付、流水、结算状态机。  
- **可审计**：关键状态迁移 **append-only 事件表或审计日志**（与文档第九章、合规方向一致）。  
- **幂等**：外部支付回调、平台 API 拉取、定时任务。

### 6.4 安全边界（gstack：`/cso` 精神，轻量版）

- 支付与提现：**最小权限**、双因素、敏感操作审计。  
- 对外 webhook：**验证签名 + 重放保护**。  
- 管理端：**RBAC** 与操作留痕。

## 7. Phase D — 开发（gstack：`/review` + 实现）

### 7.1 Epic  backlog（与需求文档章节映射）

| 优先级 | Epic | 需求文档章节 | 说明 |
|--------|------|----------------|------|
| P0 | **CPM 口径与展示** | 一 | 定义有效曝光、计费口径、展示延迟说明 |
| P0 | **结算与资金流水** | 二 | 状态机、对账、失败重试、人工介入入口 |
| P1 | **KOL 检索与筛选** | 三 | 先规则/标签，再向量/大模型 |
| P1 | **内容与合规审核** | 四 | 人工 + 规则 + 异步 AI 评分 |
| P2 | **频次限制** | 五 | 滑动窗口、按平台差异化 |
| P2 | **邀请与增长** | 六 | 邀请码、反作弊 |
| P1 | **预算控制** | 七 | 冻结、分配、告警 |
| 持续 | **安全合规** | 九 | 随业务区域扩展 |

### 7.2 本阶段「开发」定义

在 gstack 语义下，**开发** = 在清晰切片下 **可合并、可回滚、可测试** 的增量。本报告不代替一次完整编码迭代，但给出 **可执行的任务拆分**，供下一迭代直接开单。
